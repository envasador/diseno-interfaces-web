## **3\. Arquitectura de información: ordenar para entender**

Cuando alguien entra en tu web o app, no espera tener que explorar a ciegas. Necesita encontrar lo que busca rápido y saber dónde está en todo momento. Para que esto sea posible, el contenido debe estar bien estructurado. Esa estructura lógica es la **arquitectura de información (IA)**.

Diseñar una interfaz no es sólo decidir cómo se ve una pantalla, sino **cómo se relacionan las pantallas entre sí**, cómo se agrupa el contenido y qué rutas puede seguir el usuario.

### **¿Qué es la arquitectura de información?**

La arquitectura de información es **organizar, etiquetar y jerarquizar contenidos** para que sean fáciles de usar y entender. Se aplica a webs, apps e incluso sistemas físicos como museos.

En diseño digital, la IA responde preguntas como:

* ¿Cuántas secciones principales hay?
* ¿Qué subsecciones dependen de cada una?
* ¿Dónde colocas ciertas funcionalidades?
* ¿Qué estructura de navegación es más clara?

Una buena IA **no se nota, pero una mala se sufre**: el usuario se pierde, no encuentra lo que busca o repite pasos innecesarios.

### **Elementos clave de la IA**

#### **Mapa de navegación (sitemap)**

Esquema que muestra cómo se relacionan todas las pantallas. Permite ver la jerarquía de un vistazo. Lo puedes hacer en Figma, Miro o hasta en papel.

#### **Agrupación por categorías**

El contenido debe estar en bloques coherentes. En una app de viajes: "vuelos", "hoteles", "actividades", "mi cuenta". Si las categorías no están claras, el usuario se confunde.

#### **Breadcrumbs (migas de pan)**

Muestran al usuario dónde está exactamente. Ejemplo: *Inicio \> Productos \> Electrónica \> Portátiles*.

#### **Menús y navegación**

La IA también define cómo se accede a cada sección: ¿menú lateral? ¿iconos abajo? ¿pestañas arriba?

### **Principios de Dan Brown para IA**

1. **Objetividad**: diseñar para el usuario, no para ti
2. **Organización**: agrupar por lógica, no por estética
3. **Navegabilidad**: el usuario siempre debe saber dónde está y cómo volver
4. **Contexto**: cada sección debe tener sentido por sí misma
5. **Trazabilidad**: permitir seguir el rastro de la navegación
6. **Consistencia**: usar la misma estructura y lenguaje en todo el sistema
7. **Economía**: no abrumar con opciones innecesarias

### **Ejemplo práctico: app de gestión de tareas**

App que permite crear tareas, organizarlas por proyectos y hacer seguimiento:

* Secciones principales: "Inicio", "Proyectos", "Tareas", "Calendario", "Cuenta"
* Dentro de "Proyectos": lista de proyectos → al entrar en uno, sus tareas
* Navegación: menú inferior con iconos \+ botón flotante para crear tareas
* El usuario debe ver claramente en qué proyecto está y poder volver sin perderse

Diseñar esta estructura antes de hacer wireframes evita improvisaciones y da coherencia al producto.

### **Actividad práctica**

**"Arquitectura para tu proyecto"**

Esta actividad es preparatoria directa para la **Fase 2** del proyecto:

1. Cada persona trabaja en la temática elegida para su proyecto intermodular
2. Crear un mapa de navegación preliminar que incluya:
    * Secciones principales para vuestra problemática específica
    * Subpantallas y funcionalidades que creéis necesarias
    * Flujo principal que seguirá vuestro usuario objetivo
3. Hacerlo en papel, Figma o Miro.
4. **Conectar con el proyecto**: Esta arquitectura será la base para la entrega de la Fase 2

**Nota importante**: No busquéis hacer algo original en esta fase. Buscad resolver el problema de forma clara usando patrones conocidos. La originalidad viene de resolver bien las cosas, no de inventar navegaciones complicadas.
