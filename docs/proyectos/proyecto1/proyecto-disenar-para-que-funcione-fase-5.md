## **Fase 5: Define tu MVP**

¿Qué es lo mínimo que necesitas para que funcione? No todo tiene que estar en la primera versión.

#### **Qué tienes que hacer:**

Llega el momento de tomar decisiones estratégicas. Tienes un prototipo que probablemente incluye muchas funcionalidades, pero no todo puede estar en la primera versión. Aquí es donde entra el concepto de MVP: separar lo que es esencial para que funcione de lo que sería bonito tener pero puede esperar.

Para definir qué entra en tu MVP, pregúntate qué es lo mínimo que necesita tu usuario para resolver su problema principal. Si estás diseñando una app de gestión de gastos, quizás el MVP sea solo registrar gastos, categorizarlos y ver un resumen básico. Las gráficas avanzadas, la sincronización con bancos o los reportes automáticos pueden venir después.

Cada decisión tiene que estar justificada estratégicamente, no por capricho. Explica por qué ciertas funcionalidades están dentro del MVP y otras fuera. Piensa en términos de riesgo, complejidad técnica, y valor para el usuario. A veces es mejor lanzar algo simple que funcione perfectamente que algo complejo a medias.

Una vez definas tu MVP, actualiza tu prototipo para reflejar únicamente esas funcionalidades. Esto no significa eliminar el trabajo anterior, sino crear una versión "v1.0" limpia que sea la base para el desarrollo técnico. También piensa en el roadmap futuro: ¿en qué orden añadirías las funcionalidades que quedan fuera?

#### **Entregable:**

La definición del MVP debe incluir una tabla clara con funcionalidades dentro y fuera del MVP junto con la justificación estratégica de cada decisión, el prototipo actualizado que refleje únicamente las funcionalidades MVP, y un roadmap básico indicando el orden de implementación de futuras versiones.
